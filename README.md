# Weave

> Code-first object abstraction over PostgreSQL. Think in nested objects, never write SQL.

Weave lets you model your data as **objects** ÔÇö nested as deep as you like ÔÇö and
translates that into a real relational schema (tables, foreign keys, indexes). It
materializes the DDL for you, and **weaves** flat rows back into object trees on
read. Underneath it's 100% relational: you get referential integrity, ACID
transactions, and the Postgres optimizer for free ÔÇö you just never see SQL.

> **Status:** pre-release / alpha. The API may still change.

## How it fits together

```
your app ÔöÇÔöÇHTTPÔöÇÔöÇÔľÂ Weave server ÔöÇÔöÇSQLÔöÇÔöÇÔľÂ Postgres
 (weave-sdk)       (container)          (Weave owns this)
```

Weave is **two things**: a **server** you run as a container
(`ghcr.io/mauro-andre/weave`), and a **typed client** you install
(`@mauroandre/weave-sdk`). Your app declares entities as code, talks to the server over
HTTP, and thinks only in objects ÔÇö HTTP, JSON and SQL all stay invisible. The server
owns the Postgres connection and the schema; your app never connects to a database and
never writes a migration.

The server also ships a **dashboard**: a visual entity designer, a data browser, API
keys, and scope editing. Code and GUI are *isomorphic* ÔÇö both produce the same thing, and
`weave gen` mirrors the server back into `.ts` files you commit.

## Why

The objectÔÇôrelational impedance mismatch forces a bad trade:

- **MongoDB** ÔÇö great object ergonomics, weak relational guarantees; embeds
  everything by default, which becomes consistency debt.
- **ORMs** (Prisma, TypeORM) ÔÇö hide SQL behind heavy layers and their own schema
  languages.
- **Query builders** (Kysely, Drizzle) ÔÇö excellent, but keep you glued to SQL and
  the row/table model.

Weave aims at the missing spot: **Postgres' relational guarantees + Mongo's object
ergonomics + zero SQL in application code.**

## The model: two relationships, both foreign keys

Everything is relational. What differs between two relationships isn't *where* the
data lives ÔÇö it's **who owns it** and **who controls its lifecycle**.

| | `owned` (composition) | `reference` (association) |
|---|---|---|
| Ownership | exclusive to the parent | independent / shareable |
| Lifecycle | dies with the parent (**cascade**) | lives on its own (**no cascade**) |
| Writes | managed by the parent | read-only from this side |
| Storage | dedicated, path-prefixed table | FK column / join table |
| Reads | automatic (part of the aggregate) | on demand (`expand` / `select`) |

## Requirements

- **PostgreSQL 13+** ÔÇö for the *server*. Ids are **UUID v7** (time-ordered,
  index-friendly), generated by Weave in TypeScript, so no native `uuidv7()` is
  required (that's Postgres 18+ only). Migrating from MongoDB? `WEAVE_ID_TYPE=objectId`
  gives you 24-hex ids instead.
- **Node 20+** ÔÇö for your app and the CLI.

## Quick start

**1. Run the server.** Point it at a Postgres and a few secrets:

```bash
docker run -p 3000:3000 \
  -e DATABASE_URL=postgres://user:pass@host:5432/db \
  -e MASTER_USERNAME=master \
  -e MASTER_PASSWORD=change-me \
  -e SESSION_SECRET=a-long-random-string \
  ghcr.io/mauro-andre/weave
```

Open `http://localhost:3000`, log in, and create an API key from the API page.

**2. Install the client:**

```bash
npm install @mauroandre/weave-sdk
```

**3. Declare entities** ÔÇö one file per entity, default-exported:

```ts
// weave/entities/post.ts
import { defineEntity, text, int4, bool, array, owned, reference } from "@mauroandre/weave-sdk";
import author from "./author.js";
import tag from "./tag.js";

export default defineEntity("post", {
  title:     text().notNull(),
  published: bool().notNull().default(false),
  views:     int4().notNull().default(0),
  keywords:  array(text()),            // text[]
  author:    reference(author),        // N:1  Ôćĺ author_id
  tags:      reference(array(tag)),    // N:N  Ôćĺ post_tags join table
  comments: owned(array({              // 1:N  Ôćĺ post_comments table (cascade)
    body:     text().notNull(),
    approved: bool().notNull().default(false),
  })),
});
```

Fields are **nullable by default** ÔÇö `.notNull()` makes one required.

A third argument declares **composite** unique constraints and indexes (each group is a
list of field names; a `reference` maps to its foreign key) ÔÇö the natural/rollup key a
single-column `.unique()` can't express:

```ts
defineEntity("registryEntry",
  { slugName: text().notNull(), stack: reference(stack) },
  { unique: [["slugName", "stack"]] },
);
```

**4. Push the schema** ÔÇö the server creates the tables:

```bash
export WEAVE_URL=http://localhost:3000
export WEAVE_KEY=<your api key>
npx weave push
```

**5. Use the typed client:**

```ts
import { createClient } from "@mauroandre/weave-sdk";
import author from "./weave/entities/author.js";
import post from "./weave/entities/post.js";

const weave = createClient({
  url: process.env.WEAVE_URL!,
  key: process.env.WEAVE_KEY!,
  entities: { author, post },
});

const a = await weave.author.create({ name: "Ada", email: "ada@x.dev" });
const p = await weave.post.create({ title: "Why Weave", authorId: a.id });

p.views;      // number  (inferred)
p.createdAt;  // Date    (revived from JSON)
```

`InferEntity<typeof post>` is inferred automatically ÔÇö `id`, `createdAt` and
`updatedAt` are managed for you.

## Writing

`create` shreds the object into rows and upserts the whole aggregate in one
transaction. You set references by id; the referenced entity is never touched.

```ts
const ts = await weave.tag.create({ label: "typescript" });

const p = await weave.post.create({
  title: "Why Weave",
  authorId: someAuthorId,        // N:1 reference, by id
  tagsIds: [ts.id],              // N:N reference, by id set
  comments: [{ body: "great!" }] // owned children, written in cascade
});
```

- `owned` children are **replaced** on write (the object is the source of truth): the
  existing children are deleted and re-inserted. So an owned child's `id` and
  `createdAt` **change on every write** ÔÇö treat owned children as values, and never
  reference one by id from elsewhere.
- `reference` is set via `<field>Id` (N:1) or `<field>Ids` (N:N) ÔÇö Weave never writes
  the target entity. **`<field>Ids` replaces the whole set**, so to add a link you send
  the full list, not just the new id.

Verbs per entity: `create` ┬Ě `createMany` ┬Ě `findOne` ┬Ě `findMany` ┬Ě `paginate` ┬Ě
`updateOne` ┬Ě `updateMany` ┬Ě `deleteOne` ┬Ě `deleteMany` ┬Ě `aggregate` ┬Ě `accumulate`.
Target rows with a bare `where` (`{ id: "x" }` Ôëí `{ id: { eq: "x" } }`); `One` takes the
first match, `Many` operates in bulk and returns `{ count }`. An update is a **merge** ÔÇö
fields you omit are preserved.

## Reading

`owned` comes back automatically. `reference` comes only when you ask for it.

```ts
const posts = await weave.post.findMany(
  { published: true },
  { expand: { author: true, tags: true } },   // follow references
);
// posts[0].comments[]  Ôćĺ owned, nested automatically
// posts[0].authorId    Ôćĺ the FK is always there
// posts[0].author      Ôćĺ the object, because you expanded it
// posts[0].tags[]      Ôćĺ N:N targets, because expanded
```

The return type **self-types by your `expand`** (`InferRead<E, X>`) ÔÇö a reference you
didn't expand doesn't exist on the type, so you never hand-write a result type and never
guess whether a field is populated.

### Filtering ÔÇö the whole tree, compiled to indexed `EXISTS`

```ts
await weave.post.findMany({
  published: true,
  views:    { gte: 1000 },                  // operators
  keywords: { has: "typescript" },          // array column
  author:   { name: { ilike: "mauro%" } },  // reference target  Ôćĺ EXISTS
  tags:     { some: { label: "postgres" } },// N:N quantifier    Ôćĺ EXISTS
  comments: { every: { approved: true } },  // owned quantifier
  or: [ /* ... */ ],                        // and / or / not
}, { orderBy: { views: "desc" } });
```

- **Operators:** `eq` (bare value), `ne`, `gt`/`gte`/`lt`/`lte`, `in`/`notIn`,
  `like`/`ilike` (strings), `isNull`.
- **Array columns:** `has`, `hasSome`, `hasEvery`, `isEmpty`, and `some` ÔÇö *any element
  matches scalar ops*, e.g. `{ scores: { some: { gte: 5 } } }`.
- **Quantifiers** (1:N owned, N:N reference): `some` / `every` / `none`.
- **Logic:** `and` / `or` / `not`, nestable at any level.
- Filtering by a reference's raw FK: `{ authorId: "..." }`.
- **Managed fields:** filter by `id`, `createdAt`, `updatedAt`. `orderBy` accepts those
  plus **nested paths** (`{ author: { name: "asc" } }`).

This `where`/`orderBy` is a single, shared query language: the same object you build by
clicking in the GUI, write in the SDK, and store in a scope ÔÇö from the click to the SQL.

### How many rows

`findMany` returns every matching row up to a **10 000** default, and tells you when it
had to cut. Raise or lower it with `limit`; for a paged UI use `paginate`:

```ts
await weave.post.findMany({ published: true });              // up to 10 000 (warns if more matched)
await weave.post.findMany({ published: true }, { limit: 50 });
const page = await weave.post.paginate({}, { page: 2, perPage: 20 });
// { docs, docsQuantity, pageQuantity, currentPage }
```

`findMany(where, { latestPer, orderBy })` returns the latest row per group
(`DISTINCT ON`).

### Reading a subset ÔÇö `select`

A read hydrates everything you own. For a deep entity in a **list**, narrow it with
`select` ÔÇö a whitelist that mirrors the tree, and prunes the object *and* the type:

```ts
const rows = await weave.post.findMany(where, {
  select: { title: true, author: { name: true }, comments: { body: true } },
});
// rows[0]: { id; title; author: { id; name } | null; comments: { id; body }[] }
rows[0].views; // ÔŁî compile error ÔÇö not selected, doesn't exist on the type
```

Anything you don't name isn't even joined. `select` also subsumes `expand` (it drives
references in the same key). It's response *shaping*, not access control ÔÇö for that, see
**[Scopes](#scopes--access-control-as-code)**.

## Aggregation

`aggregate` rolls rows into numbers ÔÇö pushed down to Postgres, returned as objects.
Accumulators (`count`/`sum`/`avg`/`min`/`max`/`distinct`/`first`/`percentile`/`histogram`)
go in `select`, alongside `groupBy` (fields, dot-paths or `timeBucket`), `having`,
`orderBy`, top-N and `facets` (many breakdowns in one pass). Expressions
(`div`/`mul`/`add`/`sub`) let you sort and filter by a derived rate server-side:

```ts
import { count, div } from "@mauroandre/weave-sdk";

const perRoute = await weave.appRequest.aggregate({
  groupBy: ["route"],
  select: {
    errors:    count({ where: { status: { gte: 500 } } }),  // { where } Ôćĺ a filtered count
    total:     count(),
    errorRate: div("errors", "total"),
  },
  orderBy: { errorRate: "desc" },   // the routes that fail most, proportionally
});
```

For a historical tier, `accumulate(key, ops)` is a mergeable upsert ÔÇö `inc`/`max`/`min`/
`setOnInsert` applied atomically in Postgres, no read-modify-write. Batch ingest is
`createMany([...])` (one transaction). Full tour:
**[weavepg.dev/docs/aggregation](https://weavepg.dev/docs/aggregation)**.

## Migrations ÔÇö `weave push`

Editing an entity file doesn't touch the database. Migrating is an explicit, gated step
that diffs all your entities at once and applies them with the same risk buckets the GUI
shows (­ččó auto ┬Ě ­čö┤ confirm ┬Ě ­ččí needs value ┬Ě ÔŤö blocked). The server is the source of
truth.

```bash
weave push                              # entities (plan Ôćĺ apply) + scopes, then re-gen locally
weave push --confirm product.legacy     # confirm a destructive drop
weave push --fill product.sku="N/A"     # backfill a new required field
weave push --rename product.name=title  # a rename: data preserved, not drop+add
weave push --no-gen                     # apply to the server but don't rewrite local files (CI)
weave gen                               # regenerate the whole weave/ folder from the server
```

Renaming a field in code is otherwise indistinguishable from drop+add ÔÇö `--rename` tells
the server it's a rename (the field's stable id is preserved), so the data survives.

`WEAVE_URL` and `WEAVE_KEY` come from the environment; the config file holds only
structural decisions:

```ts
// weave.config.ts
import { defineConfig } from "@mauroandre/weave-sdk";
export default defineConfig({ dir: "app/weave" }); // optional ÔÇö defaults to "weave"
```

To converge the schema at boot instead of from a terminal, `pushAll({ url, key,
entities, scopes })` is the same project push as a plain function.

### `weave gen` ÔÇö the server back into code

You can author entities and scopes in the GUI *or* in code, mixed. `weave gen` is a
one-way mirror: it fetches the server's current state and **replaces** `weave/entities/`
and `weave/scopes/` ÔÇö entity files (each field carrying its stable `$id`), scope files,
the re-export barrels, and a ready-to-use client. Everything is readable `.ts`, meant to
be committed.

```
weave/
  entities/  post.ts ┬Ě author.ts ┬Ě index.ts
  scopes/    storefront.ts ┬Ě index.ts
  index.ts   # export const weave = createClient({ url: WEAVE_URL, key: WEAVE_KEY, entities })
```

> Those two folders are **rebuilt from scratch** on every `gen` ÔÇö and `weave push` runs
> `gen` at the end unless you pass `--no-gen`. Never keep hand-written code in them.

Code-first round-trip: write a field id-less Ôćĺ `weave push` (the server mints the id) Ôćĺ
`weave gen` (rewrites the file with `.$id(...)`). From then on, renames are safe. The
generated client reads `WEAVE_URL`/`WEAVE_KEY` at runtime (no secret committed) and holds
the god-mode key, so it is **server-side only**.

## Scopes ÔÇö access control as code

A scope shapes access per entity: which **verbs**, which **rows**, which **fields**. You
write it against the entity object; it's stored by stable field-id (rename-proof) and
enforced on the server.

```ts
import { defineScope, scopeRule } from "@mauroandre/weave-sdk";
import product from "../entities/product.js";

export default defineScope("storefront", [
  scopeRule(product, {
    verbs:  ["read"],
    where:  { company: { id: { eq: { param: "companyId" } } } }, // row filter, with a request param
    fields: { exclude: ["cost"] },                               // projection
  }),
]);
```

Param names are **inferred** from the rules, so narrowing a client requires them, typed:

```ts
const store = weave.as(storefront, { companyId: ctx.user.companyId });

await store.product.findMany();     // only the rows the scope allows, without `cost`
await store.product.create(/*ÔÇŽ*/);  // verb not granted Ôćĺ WeaveScopeError (403)
```

The row filter governs writes too: a `create` or `update` whose resulting row would fall
outside it is rejected ÔÇö a scope can never write into another tenant.

For a multi-tenant server, `createScopedClient(weave)` resolves the active scope from an
`AsyncLocalStorage` context, so services just call `scopedWeave.*` with no plumbing.
Outside a `runAs` it **denies** rather than falling back to god access.

> **The API key is the trust boundary.** Without `weave.as`, it is god-mode ÔÇö a scope is
> a restriction the key-holder opts into, not a wall against it. Keep the key
> server-side, and derive scope params from the authenticated principal, never from
> request input.

Errors come back **typed** ÔÇö `WeaveScopeError` (403), `WeaveValidationError` (400),
`WeaveNotFoundError` (404), `WeaveAuthError` (401) ÔÇö never a raw SQL stack.

## Repo layout

A monorepo. **`@mauroandre/weave-sdk`** is the only published package; `core` (pure
entity builders + IR + type inference) is bundled into it at publish time, and the
server ships as a container image, not a library.

```
packages/core/   entity builders ┬Ě IR ┬Ě type inference (zero runtime deps)
packages/sdk/    the published client + CLI
app/             the server: engine, HTTP API, dashboard
site/            weavepg.dev ÔÇö docs
agent/           AGENTS.md ÔÇö rules for AI agents building with Weave
```

## Design references

Weave's architecture (object on top, relational underneath, a compiler in the
middle) is validated by **Gel / EdgeDB**. The "type as object" design and nested
reads are inspired by **Drizzle `pg-core`**; PGÔćĺTS mappings by **Kanel / pg-types**.
None are dependencies.
